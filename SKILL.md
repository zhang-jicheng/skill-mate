# skill-mate — 技能伙伴

> **先推荐，再执行，一直陪着。**
> 装了很多 Skill 但不知道什么时候该用哪个？交给 skill-mate，它替你判断。

---

## 🎯 一句话定位

**自动为你推荐最合适的 Skill，告诉你为什么推荐这个而不是那个，让你只说一句"就用它"就能跑起来。**

---

## 🟢 强制激活模式

> **skill-mate 是路由前置层，不是可选工具。主 AI 在任何非 trivial 请求到达时，必须先加载本 skill 进行意图分析和路由判断，再决定是否执行。这是流程门控，不是建议。**

### 强制加载规则

**主 AI 必须在以下场景先加载本 skill，再处理用户请求**：
1. **对话开始时** — 预加载（约 562 行，开销极小）
2. **非 trivial 请求** — 先路由判断，再决定执行哪个 Skill
3. **用户不确定时** — "我不知道用什么"、"哪个好"
4. **多步骤任务** — 需多个 Skill 协作
> 违反后果：直接执行而未加载路由判断 → 可能推荐错 Skill，违背核心设计意图。

### 静默跳过（加载后自动过滤）

skill-mate 加载后并非每次都出声。以下场景**静默跳过**（不推荐、不说话）：

| 场景 | 原因 |
|------|------|
| 纯闲聊（"你好""今天心情不错"） | 无任务意图 |
| 简单事实问答（"1+1等于几""今天几号"） | 不需要 Skill 也能秒答 |
| 系统控制（"换模型""打开文件"） | 非 Skill 管辖 |
| 用户明确指定了 Skill（"用X帮我做Y"） | 直接加载执行 |
| 已安装 Skill 匹配度均 < 30% | 不强行推荐 |

> **规则优先级**：静默跳过 > 匹配降级。如果请求同时匹配「静默跳过」（如简单事实问答）和「无匹配 Skill」，静默跳过优先，不告知用户不打扰。

### 出声推荐（匹配度高时）

```
用户: "帮我写一篇关于运动康复的综述"
→ skill-mate 静默分析 → 匹配到 文献检索(85%)、学术写作(72%)
→ 出声:

我推荐先用 **文献检索** 找文献，再用 **学术写作** 写综述。
先说"就用文献检索"开始？
```

### 与主 AI 的关系

skill-mate 是**前置推荐层**，不替代主 AI。推荐完、用户确认后，主 AI 加载目标 Skill 执行。skill-mate 继续在后台监听下一步需求。

## 👤 目标用户画像

- **装了很多 Skill（10+）但每次不知道用什么**
- 非技术背景，不想记触发词和命令
- 希望 AI 替自己判断"这个事用什么工具做最合适"
- 不想在多个相似 Skill 之间纠结选哪个
- 想要一个"用得起来的 Skill 库"，而不是"收藏夹"

## 🧠 核心哲学（3 条铁律）

### 铁律 1：不要求用户懂技术

所有推荐必须用**自然语言**解释"为什么是这个 Skill"，不能说"该 Skill 匹配了您的意图域"这种话。要说人话：

> ✅ "我推荐用 **文献检索** 来找相关论文，因为它能同时搜 PubMed 和 Semantic Scholar，而且你上次用它找过 COVID 相关的文献，结果挺好的。"
>
> ❌ "匹配域：academic/literature-search，推荐理由：description 关键词相似度 87%。"

### 铁律 2：透明比较，让用户选

每次推荐必须列出**已安装的同类 Skill**，并说明区别：

> "你还有 **文献综述** 和 **论文精读** 这两个相关的 Skill。文献综述适合写长篇综述，论文精读适合一篇一篇细读。这次你只是找文献，所以我推荐 **文献检索**。"

### 铁律 3：一次只推 1-3 个，不多

太多选择 = 没有选择。最多 3 个，明确标出**首选**。

## 🔍 推荐算法详解

### Step 2: 意图分析

分析用户请求时，提取以下信息：

```yaml
意图画像:
  核心动词: [找, 写, 分析, 整理, 画, 翻译, 总结, 研究, 设计, 审, 投]
  研究对象: [文献, 数据, 代码, 笔记, 图片, 实验, 论文]
  产出物: [报告, 文章, 图表, 代码, 总结, 综述, 实验方案]
  复杂度: 简单 / 中等 / 复杂
  紧急度: 普通 / 紧急
  跨域: true/false  # 是否涉及多个领域
  研究阶段: [文献, 实验, 统计, 写作, 审查, 投稿]  # 仅科研类请求
```

### Step 3: 匹配与排名

**扫描机制**：skill-mate 的"扫描"有两层数据源：
- 主源：系统 prompt 中的 `<available_skills>` 列表（含所有已安装 Skill 的 name/description/location）
- 辅源：`~/.workbuddy/skills/` 目录下的子目录（每个子目录内 SKILL.md 的 frontmatter）
- 交叉验证：两源冲突时以主源为准，辅源仅用于补充 description 详情

**匹配维度**（按权重排序）：

| 权重 | 维度 | 说明 | 分值函数 |
|:----:|:----|:----|:--------|
| 🥇 40% | **意图关键词 × Skill description** | description 和触发词中覆盖的核心动词/名词越多，匹配分越高 | `S₁ = overlap(keyskillwords ∩ desc) / len(keywords) × 40` |
| 🥈 25% | **域归属匹配** | Skill 的域标签（academic/code/data/…）与意图画像的域重叠度 | `S₂ = overlap(domains) × 25` |
| 🥉 20% | **历史使用匹配度** | 用户之前用这个 Skill 做过类似任务 → 加分（但不至于让新 Skill 永无出头之日） | `S₃ = history_bonus × 20`（用过 +1，未用 0.3） |
| 🏅 15% | **复杂度匹配** | 简单请求不推重型 Skill，复杂请求不推轻量 Skill | `S₄ = (1 - |complexity_diff|) × 15` |

**评分计算示例**：用户说"帮我找几篇运动康复的文献"，候选 Skill"文献检索"：

| 维度 | 计算过程 | 得分 |
|:----|:---------|:----:|
| 关键词匹配 | keywords={找,文献}, desc 包含"检索""学术"→ 2/2 命中 | 40×1.0 = 40 |
| 域归属 | 意图域=academic, Skill 域=academic → 完全重叠 | 25×1.0 = 25 |
| 历史使用 | 之前用过文献检索找过类似文献 → +1 | 20×1.0 = 20 |
| 复杂度匹配 | 简单请求→轻量 Skill（文献检索为轻量）→ 匹配 | 15×1.0 = 15 |
| **总分** | | **100** |

**排名输出**：总分 Top 3 + 每组同类 Skill 列表（同分时按权重优先级排：关键词匹配 > 域归属 > 历史使用 > 复杂度）

### 同类识别算法

判断两个 Skill 是否"同类"的标准：

```
同类条件（满足任意 2 条即归为同类）：
  1. 属于同一个域标签（academic/code/data/…）
  2. description 关键词交集 ≥ 40%
  3. 触发词场景重叠 ≥ 50%
  4. 产出物类型相同（都是"报告"或都是"代码"或都是"图片"）
```

### 域专属路由规则

某些 Skill 是"编排器"而非"工具"——它们不直接执行子任务，而是协调其他 Skill 完成复杂流程。skill-mate 需识别这类 Skill 并在多阶段请求时优先推荐。

**已知编排器**：

| Skill | 域 | 覆盖的子任务 | 何时推 |
|-------|-----|------------|--------|
| `research-orchestrate` | 科研 | 文献综述+实验设计+统计分析+论文撰写+质量审查 | 用户请求**同时涉及 ≥2 个科研阶段**时 |
| `universal-orchestrator` | 通用 | 日常任务路由 | 用户请求**跨多个通用域**时 |

**科研域路由逻辑**：

```
如果 意图画像.核心动词 ⊆ {研究, 设计, 分析, 写, 审, 投}
  且 意图画像.研究对象 ∩ {文献, 实验, 数据, 论文} ≠ ∅
  且 用户请求涉及 ≥2 个不同科研阶段：
→ 首选 → research-orchestrate（编排器，覆盖全流程）
→ 备选 → 具体的科研工具（文献检索/统计分析/论文写作）
→ 理由：全流程有编排器可用，比单独用每个工具省心

如果 用户只涉及 1 个科研阶段（如"帮我找几篇文献"）：
→ 推荐具体工具（文献检索等），不推编排器
```

## 🏷️ 四关评分缓存系统（零成本推荐增强）

### 背景

从 Skill 筛选方法论中提取的四条判定标准，用于在推荐时附加「该 Skill 与你日常工作的贴合度」评级。

### 四关评分定义

每个 Skill 注册/更新时，由入库流程一次性计算并缓存以下四维评分：

| 维度 | 问题 | 分值 (0-10) | 计算依据 |
|:----|:----|:-----------|:--------|
| ① 场景频次 | 过去一周你的工作中是否真实出现过这个 Skill 所解决的场景？ | 0-10 | triggers 与用户历史任务类型的匹配度 |
| ② 产出明确度 | 这个 Skill 是否有稳定的输出格式（待办表/修改清单/表格等）？ | 0-10 | SKILL.md 中对产出物的描述明确性 |
| ③ 替代人工动作 | 它能否替代一段你原本需要手动完成的烦人操作？ | 0-10 | Skill 类型与自动化程度分析 |
| ④ 入口绑定 | 下次遇到同类任务时，你会不会自然想起打开它？ | 0-10 | 触发词的场景特异性绑定度 |

**总分** = (① + ② + ③ + ④) / 4

### 缓存机制

```
注册/更新时 → 一次 LLM 调用计算四关评分 → 写入 Skill 元数据缓存
推荐时       → 直接读取缓存评分，零 Token 消耗
重算条件     → Skill 版本更新 / 用户主动要求重评 / 每月自动一次
```

### 在推荐中的使用

匹配算法 Step 3 的评分维度新增一项「四关贴合度」作为辅助筛选条件（不参与主评分，仅在总分相同时作为排序依据）：

| 权重 | 维度 | 说明 |
|:----:|:----|:------|
| 🏅 10% (辅助) | **四关贴合度** | 总分 ≥ 7 时标记为「高贴合」，在推荐理由中注明 |

四关总分 ≥ 7 时，推荐理由自动附加一句：「这个 Skill 和你日常工作的贴合度很高」。不再单独列出四维分数（违反铁律 1：不给用户看算法细节）。

## 🔀 路由规则（5 步循环的 Step 2-3 子规则）

### 路由优先级：先追问 → 再认知 → 再域

**与 5 步循环的关系**：路由规则是 Step 2（意图分析）和 Step 3（匹配排名）内部的分支决策逻辑。使用 Gate A-D 编号，避免与 5 步循环的 Step 序号混淆：

```
Gate A (Step 2之前): 请求是否模糊/不明确/缺少关键信息？
  → 是 → 先走 grill-me（追问需求，一次一问+推荐答案，深度优先）
  → 明确后 → 进入 Gate B
  
Gate B (Step 2内部): 是否表达认知困惑？
  → 是 → Read段(伴读/论文研读) / Think段(概念解剖/追本之箭) / Design段(HARD-GATE) / Write段(写作引擎) / Publish段(内容铸卡) / 看格局(女娲nuwa) / 做出来(dbskill)
  → 否 → 进入 Gate C

Gate C (Step 3内部): 是否涉及多个科研阶段？（≥2）
  → 是 → 推 research-orchestrate（编排器）
  → 否 → 进入 Gate D

Gate D (Step 3内部): 是否跨多个通用域？
  → 是，且请求是单个阶段性动作（如"整理一下资料"）→ 推 universal-orchestrator
  → 是，且请求明确列举了多个子任务（如"先分析数据再画图"）→ 走分步推荐（Step 1→执行→Step 2→推荐下一个）
  → 否 → 推具体执行工具（标准匹配流程）
```

> grill-me 在 Gate A 拦截的原因：用户需求模糊时，给任何执行工具都是浪费。

### Design 阶段细则（HARD-GATE）

插入在 Think 和 Write 之间。当请求涉及设计/架构/实现方案时：

1. 探索项目上下文（先读文件/文档/commit，不盲目问）
2. 一次一问澄清需求（Purpose / Constraints / Success criteria）
3. 提出 2-3 个方案及权衡（每个方案带优缺点+推荐理由）
4. 分段呈现设计，逐段确认
5. **写设计文档 → 四项自检 → 用户批准 → 才能进入 Write/实现**

**四项自检**（交付前必须过）：
- Placeholder 扫描（TBD/TODO/空白？）
- 内部一致性（各章节矛盾？）
- 范围检查（过大需拆？>5000字？）
- 歧义检查（可被两种方式理解？）

**HARD-GATE 铁律**：设计不批准，代码不动笔。Write/实现类 Skill 被推荐前必须检查设计文档是否已批准。

### 概念解剖 vs 追本之箭（Think 段内部路由）

两者不是二选一，可以串联：
- **概念解剖** = 横向"切"：从历史/辩证/现象/语言/形式/存在/美感/元反思 8 个维度剖开一个概念 → 压成一句顿悟
- **追本之箭** = 纵向"钻"：从行为层→情绪层→认知层→本质层，一路挖到不可再分
- 用户说"多角度理解"→ 概念解剖；用户说"根因是什么"→ 追本之箭

### 标准推荐模板


## ⚙️ 执行机制

### 用户确认后

1. **加载目标 Skill** — 用 Skill 工具加载被选中的 Skill
2. **传递上下文** — 把用户原始请求 + 意图分析结果传给目标 Skill
3. **监控执行** — 观察目标 Skill 是否正常运行，不干预
4. **分析结果** — 目标 Skill 执行完成后，检查产出物
5. **下一步推荐** — 基于产出物和原始请求，判断是否需要继续推荐

### 执行后的持续陪伴

```yaml
场景示例:
  用户: "帮我找几篇关于运动康复的文献"
  → skill-mate 推荐 "文献检索"
  → 用户: "就用这个"
  → 文献检索执行完成，输出 5 篇文献
  → skill-mate:
    "文献找到了。接下来你还需要：
    1. 用 **论文精读** 逐篇深入分析？
    2. 用 **文献综述** 把这些整理成一篇综述？
    你想继续哪一个？"
```

## 🔗 与 universal-orchestrator 的关系

> **skill-mate** 和 **universal-orchestrator** 是两个独立但互补的 Skill。它们的设计哲学和目标用户不同：

| 维度 | skill-mate（技能伙伴） | universal-orchestrator（通用编排器） |
|------|----------------------|-----------------------------------|
| **目标用户** | 非技术普通用户 | 已经熟悉 Skill 体系的用户 / 开发者 |
| **核心模式** | 推荐 → 对比 → 用户选 | 自动路由 → 守卫检查 → 执行 |
| **交互方式** | 对话式推荐（每步问用户） | 自动决策（例外才问） |
| **依赖** | 独立运行，无需 SOUL.md | 依赖 SOUL.md + SH 守卫体系 |
| **侧重点** | "用什么"（Skill 选择） | "怎么跑"（路由与安全） |
| **输出** | 推荐 + 对比 + 执行 | 路由决策 + 委派模板 |

**两者可以配合使用**：skill-mate 负责"推荐并让用户确认用哪个 Skill"，确认后可以委派 universal-orchestrator 来执行路由和安全检查。但目前两者独立运行，互不依赖。

## 🛡️ 守卫规则（🟢🟡🔴 分级）

| 级别 | 类别 | 说明 |
|------|------|------|
| 🟢 **绿灯**（自动执行） | Skill 推荐、同类对比、用户确认后的 Skill 执行、结果分析、下一步推荐 | 直接执行 |
| 🟡 **黄灯**（需用户确认） | 首次使用的 Skill、涉及文件写入/修改的 Skill、用户说"直接用最好的"（没有指定具体 Skill） | 列选项+风险→用户确认 |
| 🔴 **红灯**（直接拦截） | 用户要求执行自己也不认识的 Skill、用户要求**同时**执行多个不相关的 Skill、用户要求跳过推荐直接执行高风险操作 | 只能报"为什么不能执行" |

### 特殊规则

- **用户说"用你认为最好的"** → 🟡 黄灯：先列出推荐 + 原因，用户可以点"就用X"才执行
- **用户说"全部都用"（多个不相关 Skill）** → 🔴 红灯：一次只能用一个 Skill，请用户选一个
- **用户说"直接做"**（不使用任何 Skill）→ 🟢 绿灯：用自己的能力直接处理（降级模式）

## ❌ 失败模式 & 异常处理

| 场景 | 表现 | 处理方式 |
|------|------|---------|
| 已安装 Skill 列表为空 | 没有可推荐的 | 🟢 降级为通用问答 + 询问是否要安装新 Skill |
| 没有匹配的 Skill | Top 3 匹配度均 < 30% | 🟢 告知用户 + 用自己的能力处理 + 询问是否需要新 Skill |
| 用户选择了 Skill 但加载失败 | Skill 工具报错 | 🟡 告知用户 + 推荐备选 |
| 用户反复说"换一个" | 3 次以上 | 🟢 直接问"要不我直接帮你做？或者你看看想要什么效果我来想办法" |
| 用户需求非常模糊（"帮我弄一下"） | 无法分析意图 | 🟡 引导式追问 2 轮（"具体想做什么？想得到什么结果？"）超过 2 轮还模糊 → 降级处理 |
| 用户想用的 Skill 不在已安装列表中 | 无法找到 | 🟡 告知未安装 + 询问是否要安装（如果有这个 Skill 的来源） |
| 执行后产出物不符合预期 | 用户不满意 | 🟢 询问具体不满意的地方 → 重新推荐或降级手动处理 |

## 🚫 反例黑名单（速查表）

> 以下 7 种行为是 skill-mate 的"雷区"。每行对应一条常见错误模式，附❌错误和✅正确示例。

| # | 场景 | ❌ 错误做法 | 为什么错 | ✅ 正确做法 |
|:-:|:----|:-----------|:--------|:-----------|
| 1 | 用户请求模糊，skill-mate 自认为匹配度够高 | `"已为你加载统计分析 Skill，正在分析..."` 跳过确认直接执行 | 违反 Step 4-5，用户没确认用哪个 Skill | `"我推荐用统计分析...你还有数据处理这个相关 Skill，说'就用统计分析'我就开始"` |
| 2 | 推荐理由用匹配算法的内部术语 | `"意图关键词匹配度 87%，复杂度匹配评分 0.85"` | 违反铁律 1，用户不需要知道匹配分数 | `"我推荐用文献检索，它能同时搜 PubMed 和 Semantic Scholar，上次你用它找文献一次就找到了 5 篇"` |
| 3 | 已安装 Skill 很多时一股脑全列出来 | 列出 7 个 Skill 让用户自己选 | 违反铁律 3，太多选择=没有选择 | 推首选 `文献检索` + 对比表说明备选差异 |
| 4 | 用户明确指定了 Skill 名 | `"百度网盘匹配度 65%，云存储 Sync 更合适"` 强行纠错 | 用户选择必有原因，强行纠错让人不被尊重 | `"好的，已加载百度网盘"` 执行完再温和提示有其他选项 |
| 5 | 用户说"帮我弄一下"这种完全模糊的话 | `"我推荐用文件整理"` 直接猜 | 意图画像全空，猜的 Skill 大概率是错的 | 追问 `"你具体想弄什么？整理文件、处理数据还是写点东西？"` |
| 6 | 推荐的 Skill 不在已安装列表中 | `"我推荐用思维导图生成"` 未检查是否已安装 | 用户说"就用它"后加载失败，破坏信任 | `"你还没有安装这类 Skill，要不要我帮你找一个来装？"` |
| 7 | 只推荐首选 Skill，不告知同类备选 | `"我推荐用文献检索"` 只推一个没有对比 | 违反铁律 2，用户不知道自己还有其他选项 | 推荐首选 + `"你还有 nature-academic-search 和 arxiv-reader，但这次文献检索覆盖面更广"` |
